This Software Requirements Specification illustrates, in clear terms, the system’s primary uses and required functionality so that the next developers can easily understand what we did and how to change anything if it is required.

1.4 Product Scope

Users can sign up to the system. Each user can create an Exam-Cohort. The user who creates an exam-cohort is the evaluator for that exam-cohort. He/She can add other users of the system as candidates to the Exam-cohort.

Within each exam-cohort, the evaluator can create one or more assessments. When creating an assessment, the evaluator must supply the 'available date-time', the 'due date-time' and specify the questions.

There can be multiple questions. For each question a mark must be specified. Questions can MCQs or micro-viva. For both the marking should be automatic. For MCQs the evaluator should mark the correct answer(s). When presenting MCQs to the candidates, the options should be randomized in order. For a micro-viva the question is supplied as a recording and a correct answer is also supplied as a recording by the evaluator. During the exam, the candidate will hear the question because the system will play the recorded question. Then the candidate will answer using his/her voice which will be recorded. The system should run a speech recognition solution on the recording and compare the answer with the pre-specified answer for a correct/incorrect assessment. For now, you can assume that the answers are going to be a maximum of 5 words. For each question, irrespective of type, the evaluator must set a maximum-time, it is the number of seconds allowed for a student to answer a question before the system moves to the next one. The order of questions should also be randomized when presenting to candidate.

The marking should be done automatically, as mentioned previously. Evaluators should be able to see/hear/read the answer of any candidate within the exam-cohort. The evaluator can also adjust the mark of any question of an assessment within the exam-cohort.

Evaluators should be able to download a spreadsheet of marks, with full breakdowns of each mark of each assessment for each student.Users should be able to do all of the above (including candidates completing assessment questions) on a web application. In addition there should be a mobile application that allows users to sign in and complete assessments. Needless to say, all data should be properly synchronized among all views.

There will be no separate sign up process. Users should be able to sign in using a Google Account.

3. External Interface Requirements

All inputs and outputs necessary for the requirements will be specified in this section. Hardware, Software and communication interfaces will also be characterized here.

3.1 User Interfaces

When the user uses the web for the first time, they will be prompted for registering by google sign up. Already registered users can select to login through a button which will bring them to a log-in page. Users should be able to login by using their email and password which he/she used in their google account.

If the user is already logged into the system, they will land on the home page. The home page will show the exam cohort. Every user can create the assessment and they will add user who can participate in this assessment and the owner can see the mark of all participant in spreadsheet.

There can be multiple questions. For each question a mark must be specified. Questions can MCQs or micro-viva. For both the marking should be automatic. 3.2 Hardware Interfaces

Neither the mobile application nor the web version has any special hardware requirements. So, the system will not have any hardware interface.

3.3 Software Interfaces

Both the mobile application and the website will communicate with the database via internet. The operating system will provide such interfaces. Both the platforms will consist of operations regarding read and write of data.

3.4 Communications Interfaces

The means of communication between the different part of the system is not important and will be handled by the operations provided by the browser or operating system.
